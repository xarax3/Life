# Control Chart

> یکی از مهم‌ترین گزارش‌های Jira برای تحلیل Cycle Time و پیش‌بینی‌پذیری (Predictability)

---

# Definition

Control Chart گزارشی است که مدت زمان انجام هر Issue (Cycle Time) را نمایش می‌دهد.

برخلاف CFD که وضعیت کل فرآیند را نشان می‌دهد، Control Chart روی **هر Issue به‌صورت جداگانه** تمرکز می‌کند.

---

# Business Purpose

Control Chart به این سوالات پاسخ می‌دهد:

- آیا زمان انجام کارها ثابت است؟
- آیا فرآیند قابل پیش‌بینی است؟
- آیا Outlier وجود دارد؟
- آیا Cycle Time در حال کاهش یا افزایش است؟
- آیا تیم پایدار کار می‌کند؟

---

# محل در Jira

```
Project
    ↓
Reports
    ↓
Control Chart
```

در Jira Software به‌صورت پیش‌فرض موجود است.

---

# ساختار نمودار

محور افقی:

```
Issue Completion Date
```

محور عمودی:

```
Cycle Time
```

هر نقطه روی نمودار:

```
یک Issue
```

را نمایش می‌دهد.

---

# مثال

```

10

+------------------------

     Time
```

هر دایره نماینده یک Issue است.

---

# خطوط مهم نمودار

Control Chart معمولاً شامل این خطوط است:

- Average Cycle Time
- Rolling Average
- Percentile (در برخی ابزارها)
- Data Points

---

# Average Line

خط میانگین نشان می‌دهد:

```
میانگین زمان انجام Issueها
```

اگر اکثر نقاط نزدیک این خط باشند،

فرآیند پایدار است.

---

# Rolling Average

میانگین متحرک برای مشاهده روند تغییرات.

مثلاً:

```
Sprint 1

8 Days
```

↓

```
Sprint 5

5 Days
```

یعنی عملکرد تیم بهبود یافته است.

---

# Outlier چیست؟

Issueهایی که Cycle Time بسیار بیشتری نسبت به سایر Issueها دارند.

مثلاً:

```
Average

4 Days
```

اما یک Issue:

```
25 Days
```

طول کشیده است.

این Issue یک Outlier محسوب می‌شود.

---

# چرا Outlier مهم است؟

زیرا ممکن است نشان‌دهنده:

- Blocked Issue
- Bug بحرانی
- وابستگی به تیم دیگر
- Approval طولانی
- مشکل زیرساخت
- کمبود منابع

باشد.

---

# Enterprise Example

یک تیم توسعه داده‌های زیر را دارد:

| Issue | Cycle Time |
|--------|-----------:|
| CRM-101 | 3 روز |
| CRM-102 | 4 روز |
| CRM-103 | 3 روز |
| CRM-104 | 21 روز |
| CRM-105 | 4 روز |

Issue شماره 104 یک Outlier است.

بررسی مشخص می‌کند:

منتظر تأیید تیم امنیت بوده است.

---

# استفاده برای Forecast

اگر اکثر Issueها:

بین

```
4 تا 6 روز
```

تکمیل شوند،

می‌توان با احتمال بالا پیش‌بینی کرد که Issue جدید نیز در همین بازه تکمیل خواهد شد.

---

# ارتباط با Cycle Time

Control Chart در واقع نمایش گرافیکی Cycle Time است.

تمام تحلیل‌های Cycle Time از این گزارش قابل استخراج هستند.

---

# ارتباط با Lead Time

Control Chart به‌صورت پیش‌فرض Lead Time را نمایش نمی‌دهد.

اما می‌توان آن را با افزونه‌هایی مانند:

- eazyBI
- ActionableAgile

ترکیب کرد.

# Best Practices

✅ Outlierها را جداگانه بررسی کنید.

✅ میانگین را به‌تنهایی ملاک قرار ندهید.

✅ روند چند Sprint را مقایسه کنید.

✅ داده‌ها را بر اساس نوع Issue تحلیل کنید.

---

# Common Mistakes

❌ حذف Outlierها بدون تحلیل

گاهی مهم‌ترین اطلاعات در همین Outlierها هستند.

---

❌ نتیجه‌گیری از چند Issue

Control Chart برای تحلیل روند نیاز به داده کافی دارد.

---

❌ استفاده برای ارزیابی عملکرد افراد

Cycle Time و Control Chart شاخص‌های فرآیندی هستند، نه فردی.

---

# Interview Questions

### Control Chart چیست؟

گزارشی که Cycle Time هر Issue را نمایش می‌دهد.

---

### مهم‌ترین کاربرد آن چیست؟

تحلیل پایداری فرآیند و شناسایی Outlierها.

---

### اگر نقاط نمودار بسیار پراکنده باشند؟

فرآیند قابل پیش‌بینی نیست و باید علت نوسان بررسی شود.

---

### اگر روند Cycle Time نزولی باشد؟

فرآیند در حال بهبود است.

# Enterprise Tips

## 1. از Percentile به جای Average استفاده کنید

میانگین ممکن است تحت تأثیر Outlierها قرار گیرد.

در بسیاری از سازمان‌ها از:

- 50th Percentile (Median)
- 85th Percentile
- 95th Percentile

برای پیش‌بینی زمان تحویل استفاده می‌شود.

---

## 2. تحلیل بر اساس نوع Issue

Cycle Time مربوط به Bugها را با Storyها ترکیب نکنید.

هر نوع Issue رفتار متفاوتی دارد.

---

## 3. تحلیل بر اساس تیم

اگر چند تیم روی یک Board کار می‌کنند، داده‌ها را به تفکیک تیم بررسی کنید.

---

## 4. تحلیل بر اساس Component

ممکن است فقط یک Component باعث افزایش Cycle Time شده باشد.

---

## 5. بررسی روند ماهانه

به جای تصمیم‌گیری بر اساس یک Sprint، روند سه تا شش ماه گذشته را تحلیل کنید.

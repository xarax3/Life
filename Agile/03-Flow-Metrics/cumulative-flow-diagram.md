# Cumulative Flow Diagram (CFD)

> قدرتمندترین گزارش برای تحلیل جریان کار (Flow Analysis) در Jira و Kanban

---

# Definition

Cumulative Flow Diagram یا **CFD** نموداری است که تعداد Issueها را در هر وضعیت (Status) در طول زمان نمایش می‌دهد.

به کمک CFD می‌توان فهمید:

- جریان کار پایدار است یا خیر.
- گلوگاه (Bottleneck) کجاست.
- WIP در حال افزایش است یا کاهش.
- Throughput مناسب است یا خیر.

---

# Business Purpose

CFD به این سؤالات پاسخ می‌دهد:

- آیا فرآیند توسعه پایدار است؟
- در کدام مرحله کارها متوقف می‌شوند؟
- آیا تیم بیش از ظرفیت خود کار پذیرفته است؟
- آیا خروجی تیم ثابت است؟

---

# محل در Jira

```
Project
    ↓
Reports
    ↓
Cumulative Flow Diagram
```

این گزارش به‌صورت پیش‌فرض در **Jira Software** برای Boardهای Scrum و Kanban (بسته به نوع پروژه و نسخه) در دسترس است.

---

# ساختار نمودار

هر رنگ نشان‌دهنده یک Status است.

مثال:

```
Done
██████████████████

Testing
██████████

Code Review
███████

In Progress
██████

To Do
████
```

محور افقی:

```
Time
```

محور عمودی:

```
Number of Issues
```

---

# نحوه خواندن CFD

فرض کنید:

```
Done

██████████████
```

به‌صورت یکنواخت در حال رشد است.

نتیجه:

تیم به‌صورت پایدار کار تحویل می‌دهد.

---

اگر:

```
Testing

██████████████████████
```

به مرور ضخیم‌تر شود،

یعنی:

Issueها در مرحله Testing انباشته شده‌اند.

---

# رنگ‌ها چه معنایی دارند؟

| رنگ | وضعیت |
|------|--------|
| آبی | To Do |
| زرد | In Progress |
| نارنجی | Review |
| بنفش | Testing |
| سبز | Done |

> رنگ‌ها بسته به تنظیمات Board ممکن است متفاوت باشند.

---

# مفهوم ضخامت هر نوار

ضخامت هر نوار برابر است با:

```
تعداد Issueهای موجود در آن Status
```

مثلاً:

```
Testing

██████████████
```

یعنی تعداد زیادی Issue در تست قرار دارند.

---

# Bottleneck چیست؟

اگر فاصله بین دو خط افزایش پیدا کند،

یعنی:

Issueها در آن مرحله جمع شده‌اند.

مثال:

```
Development

██████

Testing

██████████████████
```

نتیجه:

QA نمی‌تواند با سرعت توسعه پیش برود.

---

# Flow پایدار

```
Done

──────────────

Testing

──────────────

Development

──────────────
```

اگر همه خطوط تقریباً موازی باشند،

فرآیند پایدار است.

---

# Flow ناپایدار

```
Testing

████████████████████

Development

█████

Done

██
```

نتیجه:

جریان کار متعادل نیست.

---

# ارتباط CFD با WIP

تمام Statusهای فعال بین:

```
To Do
```

و

```
Done
```

مجموعاً WIP را تشکیل می‌دهند.

اگر ضخامت این بخش زیاد شود،

WIP نیز افزایش یافته است.

---

# ارتباط CFD با Throughput

اگر نوار **Done** با شیب ثابت رشد کند:

```
Done

/

/

/

/
```

Throughput پایدار است.

اگر شیب کاهش یابد،

سرعت خروجی تیم کاهش یافته است.

---

# ارتباط CFD با Cycle Time

افزایش ضخامت نوارهای میانی معمولاً به معنی افزایش:

- WIP
- Waiting Time
- Cycle Time

است.

---

# Enterprise Example

یک تیم تجارت الکترونیک CFD زیر را مشاهده می‌کند:

```
Testing

██████████████████
```

```
Done

████
```

تحلیل:

QA گلوگاه شده است.

راهکار:

- افزایش ظرفیت QA
- تست خودکار
- تقسیم Storyها
- Pair Testing

# مزایا

✅ شناسایی سریع گلوگاه

✅ مشاهده روند کار

✅ تحلیل WIP

✅ بررسی پایداری فرآیند

✅ مناسب Kanban

---

# محدودیت‌ها

❌ علت گلوگاه را مشخص نمی‌کند.

❌ کیفیت توسعه را نشان نمی‌دهد.

❌ نیاز به Workflow صحیح دارد.

---

# Best Practices

✅ وضعیت‌های Workflow را ساده نگه دارید.

✅ از Statusهای غیرضروری اجتناب کنید.

✅ CFD را به‌صورت هفتگی بررسی کنید.

✅ تغییرات ناگهانی در ضخامت نوارها را بررسی کنید.

---

# Common Mistakes

❌ تمرکز فقط روی نوار Done

همه نوارها باید تحلیل شوند.

---

❌ تغییر مداوم Workflow

باعث سخت شدن تحلیل CFD می‌شود.

---

❌ افزایش WIP بدون کنترل

در CFD به‌سرعت قابل مشاهده خواهد بود.

---

# سناریوهای متداول

## سناریو 1

نوار Review ضخیم می‌شود.

احتمالاً:

- Reviewer کم است.
- Pull Requestها دیر بررسی می‌شوند.

---

## سناریو 2

نوار Testing ضخیم می‌شود.

احتمالاً:

- کمبود QA
- تست دستی زیاد
- محیط تست آماده نیست.

---

## سناریو 3

نوار Done تقریباً ثابت مانده است.

احتمالاً:

- Throughput کاهش یافته.
- تیم Block شده است.
- Release متوقف شده است.

---

شناسایی Bottleneck و تحلیل Flow.

---

احتمال وجود گلوگاه در فرآیند تست.

---

### اگر خطوط تقریباً موازی باشند؟

فرآیند پایدار و قابل پیش‌بینی است.

# نکات Enterprise

در سازمان‌های بزرگ، CFD معمولاً به تفکیک موارد زیر تحلیل می‌شود:

- هر تیم توسعه
- هر محصول
- هر سرویس
- هر Component
- هر Value Stream

این کار کمک می‌کند گلوگاه‌ها دقیق‌تر شناسایی شوند.

همچنین بسیاری از تیم‌ها CFD را با **DORA Metrics** (Deployment Frequency، Lead Time for Changes، Change Failure Rate و MTTR) ترکیب می‌کنند تا تصویری کامل از عملکرد تیم توسعه و عملیات به دست آورند.
